# TypeScript 从入门到高级

这是一套按学习路径设计的 TypeScript 中文专栏。它不是零散语法笔记，也不是 API 速查表，而是一条完整的理解路径：先回答“为什么要用 TypeScript”，再回答“如何用类型系统表达业务”，最后进入“如何在真实工程中设计稳定、可维护、类型安全的系统”。

如果你之前学过 JavaScript，但每次一看到 `keyof`、`infer`、泛型约束就头疼，这套文章会比传统教程更适合你。因为这里的重点不是堆语法点，而是把每个知识点放进真实问题里讲清楚。

## 这套专栏适合谁

- 已经会 JavaScript，准备系统补 TypeScript 的前端开发者
- 正在使用 React、Vue、Node.js、NestJS、Next.js，但对类型系统理解还比较碎片化的工程师
- 想从“会用 TS”进入“能设计类型”的开发者

## 推荐阅读方式

最好的方式不是一路往下滑，而是按编号顺序阅读，并把文章里的例子亲手敲一遍。TypeScript 很容易产生一种错觉：看懂了，等于会了。实际情况通常是，看懂文章只完成了 30%，真正把类型写顺手，必须经过亲手建模、报错、修正、再建模的过程。

如果你每天能投入 40 分钟到 1 小时，可以按 14 天读完一轮；如果你希望连例子、练习和自己的延伸实验一起做，建议按 28 天完成，节奏会更扎实。

## 一条更实用的学习建议

前 1 到 6 篇解决“看得懂、写得出来”；7 到 10 篇解决“为什么复杂类型可以这样工作”；11 到 14 篇解决“怎么把 TS 真正用进工程里”。

如果你是纯前端开发者，建议把所有示例都放进一个小项目里反复实验。如果你是 Node.js 或全栈开发者，尤其要重视第 12 到 14 篇，因为那几篇最接近真实团队中的接口设计、模块边界和工程配置。

## 这套专栏的使用方式

- 第一轮：只读正文，建立整体地图。
- 第二轮：照着示例手写一遍，观察编辑器如何提示。
- 第三轮：把你自己项目里的某个模块拿来重写类型。

你真正的进步，不会出现在“读完了”那一刻，而会出现在“我开始能把项目里的混乱结构重新建模”那一刻。

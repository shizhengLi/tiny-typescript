# 基础类型与类型推断

学习 TypeScript 时，很多人会急着进入泛型、工具类型、`infer` 这些“高级语法”。但如果基础类型和类型推断没有真正理解，你后面看到的很多东西都会像黑魔法。因为 TypeScript 类型系统最底层的逻辑，恰恰建立在一个非常朴素的问题上：**一个值到底是什么，编译器又是怎么知道的。**

这一篇的目标，是把最基础但最关键的一层打稳。你需要理解的不只是“有哪些类型”，更是“TypeScript 什么时候自动推断、什么时候需要你明确标注、什么时候应该尽量避免滥用宽泛类型”。

## 基础类型不是背单词，而是建立边界意识

TypeScript 常见的基础类型包括：

- `string`
- `number`
- `boolean`
- `null`
- `undefined`
- `bigint`
- `symbol`

示例很简单：

```ts
const username: string = "alice";
const age: number = 18;
const isAdmin: boolean = false;
```

这些写法看起来像语法入门题，但它们真正重要的地方在于：你开始不再把变量看成一个随时能塞进任意值的容器，而是一个有边界的值。

比如“用户名”就是字符串，“年龄”就是数字，“是否管理员”就是布尔值。这听起来显然，但很多 JavaScript 代码的问题，恰恰就出在这种边界没有被明确表达。

## `null` 和 `undefined` 的存在感比你想得更强

初学 TypeScript 时，不少人会低估 `null` 和 `undefined`。但在实际项目里，空值问题几乎是最常见的运行时错误来源之一。

例如：

```ts
function printLength(value: string | undefined) {
  console.log(value.length);
}
```

这段代码会报错，因为 `value` 可能是 `undefined`。如果你开启了严格模式，TypeScript 会强迫你正视这个问题，而不是等程序上线后再发现。

这也是为什么很多团队会把 TypeScript 看成“空值治理工具”。它不是只是在帮你写类型，它是在帮你正面处理原本被忽略的边界。

## 类型推断才是 TypeScript 真正的效率来源

很多初学者以为 TypeScript 的用法就是“每个变量后面都补上类型”。事实上，真正写得好的 TypeScript 代码通常并不啰嗦，因为编译器会替你完成大量推断工作。

```ts
const title = "TypeScript 入门";
const price = 99;
const published = true;
```

这里：

- `title` 会被推断为 `string`
- `price` 会被推断为 `number`
- `published` 会被推断为 `boolean`

所以一个非常重要的认知是：**TypeScript 不是要求你手动给一切写类型，而是让你在必要的地方表达约束，在不必要的地方利用推断节省噪音。**

## `let` 和 `const` 对推断结果有影响

很多人忽视这一点，但它非常重要。

```ts
let status1 = "success";
const status2 = "success";
```

`status1` 通常会被推断成 `string`，因为 `let` 表示后续可能被重新赋值；而 `status2` 会更精确，往往推断成字面量类型 `"success"`，因为 `const` 不允许被重新赋值。

这件事看起来很小，却和后面大量高级类型能力相关。字面量推断越精确，TypeScript 的分析能力就越强。

## `any` 很方便，但代价极大

```ts
let value: any = 123;
value = "hello";
value.toUpperCase();
value.notExist();
```

只要用了 `any`，TypeScript 基本上就退出了这段代码的保护工作。你可以做几乎任何操作，编译器都不会阻止你。

`any` 不是完全不能用，但它应该被当作“兜底逃生门”，而不是常规写法。尤其是在团队项目里，`any` 一旦蔓延，后续所有类型推导都会被污染。

一个很现实的判断标准是：只要这个值未来还会被继续传递、继续调用、继续重构，就不要轻易用 `any`。

## `unknown` 是更安全的“不确定”

当你真的不知道一个值是什么类型时，优先考虑 `unknown`：

```ts
let data: unknown = "hello";

if (typeof data === "string") {
  console.log(data.toUpperCase());
}
```

`unknown` 的价值在于它会强迫你先判断、再使用。这个顺序非常重要，因为真实世界里很多不可信数据都来自：

- 接口返回
- 用户输入
- 本地存储
- 第三方库
- JSON 解析结果

这些值在进入你系统的那一刻，并不自动安全。用 `unknown`，比用 `any` 更符合事实。

## `never` 不是玄学，它代表“不可能到达”

`never` 常被初学者看成难懂的特殊类型，其实它的含义很简单：这个值根本不会出现。

```ts
function throwError(message: string): never {
  throw new Error(message);
}
```

它通常出现在几种场景里：

- 函数永远抛错
- 死循环
- 联合类型穷尽检查

到了后面你会发现，`never` 在判断“某个分支是否已经被完全覆盖”时非常有用。也就是说，它不是为了炫技存在的，而是一个很实用的安全信号。

## `void` 也值得顺手搞清楚

```ts
function logMessage(message: string): void {
  console.log(message);
}
```

`void` 表示函数不关心返回值，或者说没有对外提供有意义的返回结果。它和 `never` 完全不同：

- `void` 是函数正常结束，但没有返回有用结果
- `never` 是函数根本不会正常结束

这一点很多人会混。

## 什么时候应该显式写类型

虽然推断很强，但不是所有地方都该偷懒。比较推荐显式标注的场景包括：

- 函数参数
- 对外暴露的函数返回值
- 公共接口、类型、类
- 复杂对象的边界定义
- 推断结果不够直观时

不太需要显式标注的场景包括：

- 一眼就能看明白的局部变量
- 右侧值本身就足够清晰时

例如下面这个局部变量，其实没必要多写一遍：

```ts
const port = 3000;
```

如果你硬写成：

```ts
const port: number = 3000;
```

通常只是增加噪音。

## 一个常见误区：把 TypeScript 学成“补标注”

很多初学者会进入一个误区：凡是没有类型注解的地方都不放心，恨不得每一行都补上。这其实没有抓住重点。

TypeScript 最有价值的地方，不是让代码到处充满注解，而是让类型信息在系统里自然流动。如果一个类型已经能被准确推断，再手动重复一遍，只会降低可读性。

真正成熟的 TypeScript 写法，往往遵循一个原则：**显式定义边界，局部细节交给推断。**

## 本文小结

基础类型是 TypeScript 的入口，类型推断是 TypeScript 的效率来源。你需要同时掌握这两件事：一方面，清楚每一种基础类型表达的是什么边界；另一方面，理解编译器已经能替你完成哪些推断，不要把 TypeScript 写成一场机械补注解。

接下来的学习中，你会越来越频繁地遇到一个问题：当一个值可能有多种合法形态时，类型系统该怎么表达？那就是下一层内容，联合类型、对象建模和类型缩小。
